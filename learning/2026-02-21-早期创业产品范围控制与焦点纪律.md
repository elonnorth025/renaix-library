# 早期创业产品范围控制与焦点纪律

> 日期：2026-02-21
> 背景：Renaix 当前产品（GC AI 员工团队）处于 PMF 验证前阶段，资源极度有限（CEO = Elon，资源提供者 = Kelvin）
> 核心问题：**如何决定做什么，以及坚定地不做什么**

---

## 为什么焦点是早期 CEO 的核心能力

创业失败的最常见原因不是「做错了」，而是「同时做了太多对的事」。

每一个「顺手加进来」的功能，都意味着：
- 多 2-4 周的开发时间
- 更复杂的销售材料
- 更难的客户 onboarding
- 更多的支持负担

**早期 CEO 的工作不是把产品做大，而是把产品做窄——窄到可以快速验证。**

Paul Graham 的建议：
> "Do things that don't scale. Serve 10 customers so well they love you, 
> rather than building scalable systems for 1000 customers who are lukewarm."

## 理论框架：Jobs-to-be-Done（JTBD）

### 核心思想

客户不是「购买产品」，而是「雇用产品完成一个任务」（Job）。

理解这个 Job，才能决定什么功能是核心（让 Job 完成得更好），什么功能是噪声（看起来相关，但换了一个 Job）。

### 三层 Job 分解

以 Renaix GC AI 员工为例：

```
功能层（Surface Job）：
"我需要有人接 GC 的电话"

执行层（Functional Job）：
"我需要不漏掉任何一个分包商进场确认，同时让业主觉得我一直在跟进"

情感层（Emotional Job）：
"我需要从每天被电话淹没的焦虑中解放出来，可以专注于拿下下一个项目"

社会层（Social Job）：
"我需要在同行中显得「专业、靠谱」——接电话很快，不会失联"
```

**只有同时解决了功能层 + 情感层，产品才会有口碑。**
功能层决定「值不值买」，情感层决定「值不值推荐给朋友」。

## 工具一：功能优先级矩阵

用两个维度评估每个功能请求/想法：

| 维度 A（X轴）：是否直接服务核心 Job | 维度 B（Y轴）：客户付费意愿 |
|---|---|
| 高度相关 = 明确完成「协调电话」这个 Job | 高意愿 = 客户会单独为此付钱，或没有它就不付钱 |
| 低相关 = 看起来有用，但和核心 Job 关联弱 | 低意愿 = 客户说「不错」但不愿意额外付钱 |

**四象限决策：**

```
                    高付费意愿
                         ↑
            ┌────────────┼────────────┐
 低 Job 相关 │  Phase 2   │  一定要做  │  高 Job 相关
            │  (等以后)  │  (NOW)     │
            ├────────────┼────────────┤
 低 Job 相关 │  永远不做  │  可能要做  │  高 Job 相关
            │            │  (验证后)  │
            └────────────┼────────────┘
                         ↓
                    低付费意愿
```

## 工具二：Scope Freeze 决策原则

### 什么是 Scope Freeze

在正式开始给第一个客户 onboard 之前，冻结 Phase 1 的功能范围。期间对所有新功能请求执行「Freeze 决策」：

**Freeze 决策树（收到功能请求时）：**

```
有客户提出新功能？
         ↓
是否 3 个以上客户独立提到相同需求？
    是 → 列入 Phase 2 Backlog，告知客户"下一个版本"
    否 → 直接谢绝，解释当前专注在核心功能
         ↓
是否不做这个功能就会失去这个客户的付费？
    是 → 认真评估（可能是核心 Job 的遗漏）
    否 → 继续冻结，感谢客户反馈
```

### GC AI 员工的 Scope Freeze 声明（供内部使用）

**Phase 1 范围（2026-02-21 冻结）：**

✅ **IN SCOPE（会做）：**
- 接听来电（业主 + 分包商 + 供应商）
- SMS / 邮件自动回复
- 分包商进场时间确认
- 业主项目进度定期更新（短信/邮件）
- 所有通话/消息记录自动存档

❌ **OUT OF SCOPE（明确不做）：**
- 任何财务功能（发票、付款、对账）
- 排期/项目管理（不与 Procore 竞争）
- 文件审核（lien waiver、pay app — 留给追销）
- 多项目并发视图 / 仪表盘
- 移动端 App

## 工具三：说「不」的话术

早期 CEO 经常面临压力（来自客户、自己的想法、市场反应），需要说不。

### 对客户说不的标准话术

**场景：** 客户要求「能不能帮我管一下项目排期？」

**错误回答：** "好的，我们研究一下！"（无边界，失焦）
**错误回答：** "不行，我们只做电话。"（太生硬，客户感觉被拒绝）

**正确回答：**
>"这个需求非常有意思，我听到很多 GC 都提到了排期问题。我们现在专注
> 在让你的协调电话和沟通问题彻底解决——这是我们能做到极致的地方。
> 排期功能已经在我们的路线图上，但我不想在还没验证核心价值之前就分散精力。
> 你愿意先用核心功能跑 2 个月，再一起看下一步需要什么吗？"

**话术要点：**
1. 承认需求真实（不否定客户）
2. 解释你的专注选择（不是"不能做"，而是"现在选择不做"）
3. 给一个未来承诺（路线图，不是谎言）
4. 拉回到当前价值（让 ta 继续向前）

## Renaix 的焦点纪律执行清单

### 每次「是否做新功能」决策前问自己：

1. **是否有 3 个以上客户独立提到？** → 没有 = 不做
2. **这个功能是否直接让「协调电话压力」消失？** → 不是 = 推后
3. **不做这个功能，当前客户会流失吗？** → 不会 = 坚持冻结
4. **我们做这个功能是因为客户需要，还是因为我们觉得「炫酷」？** → 炫酷 = 一定不做

### 焦点的代价（要有心理准备）

- 有些潜在客户会因为「功能不够多」而放弃你 → **这是好事**（过滤掉不适合的客户）
- 你会觉得产品「太简单」，不好意思收高价 → **错觉**（简单 = 易用，客户更爱）
- 竞争对手会有更多功能 → **没关系**（在你专注的细分 Job 上，你比他们好 10 倍）

## 理论补充：帕金森定律与截止日

> 帕金森定律：工作会膨胀到填满所有可用时间。

同理：**产品功能会膨胀到填满所有可用资源。**

解法：**人为制造约束。**
- 设定不可移动的 Demo 日（对外承诺）
- 在 Demo 日之前把 Scope 冻结成最小可演示集合
- 宁可砍功能，不可推迟演示

**对 Renaix 的应用：**
在第一个发现访谈完成后（预计 3 月内），立刻设定一个「对真实 GC 客户 Demo 的日期」，倒推决定哪些功能必须在那天前做完。这个日期不能推。

---

> **CEO 今日收获：** 焦点不是限制，是武器。专注的产品比全功能的竞争对手更容易被记住、更容易销售、更容易改进。Phase 1 = 3 个 Agent，做到极致，其他都是 Phase 2。
