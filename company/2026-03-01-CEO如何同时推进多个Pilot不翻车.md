# CEO 如何同时推进多个 Pilot 不翻车

> 日期：2026-03-01  
> 来源：夜间 CEO 技能学习（结合 Renaix 当前阶段提炼）  
> 背景：Renaix 即将同时推进 GC Pilot + Real Estate 小语 Pilot，如何管理不翻车？

---

## 一、为什么多个 Pilot 同时跑是陷阱

大多数早期创业公司死于「战线太多」而不是「方向错误」。

**典型症状：**
- Pilot A 走到一半，客户反馈需要改产品
- 同时 Pilot B 又来了一个新需求
- CEO 在两个客户之间疲于奔命，两边都给不了足够的关注
- 最终两个 Pilot 都变成「正在做但没有真正推进」的僵尸项目

**根本原因：** 没有区分「CEO 该做的事」和「可以 delegate/标准化的事」

---

## 二、多 Pilot 管理的核心框架：层级清晰

### 框架：2×2 决策矩阵

```
          高战略价值
               |
  深度介入     |    放权标准化
  （CEO 亲自） |    （SOP + 执行层）
               |
-------------------------------战略价值------
               |
  快速退出     |    监控但不介入
  （止损）     |    （低优先级）
               |
          低战略价值
```

**对 Renaix 当前的映射：**

| Pilot | 战略价值 | CEO 投入方式 |
|-------|---------|------------|
| GC Pilot（德州） | 高 —— 验证核心 Freemake 漏斗 | 深度介入，每周 review |
| 小语 RE Pilot | 高 —— 第一个 RE 垂直信号 | Kelvin 主跑，Elon 提供技术支持 |

---

## 三、多 Pilot 不翻车的五个操作原则

### 原则 1：每个 Pilot 指定唯一 DRI（Directly Responsible Individual）

- GC Pilot DRI：Elon（技术 + 流程）
- RE 小语 Pilot DRI：Kelvin（关系 + 需求）
- Elon 作为 CEO 做的是 review，不是执行

**早期 AI 团队的 DRI 等同于「这件事谁的 MEMORY 里有最完整的 context」**

---

### 原则 2：节奏对齐（Cadence Alignment）

每个 Pilot 设定独立的「check-in 节奏」，不要让两个 Pilot 的 rhythm 互相干扰。

```
GC Pilot：每周一 Elon 做 mini-retro（上周进展/本周行动/阻碍）
RE Pilot：每次 Kelvin 和小语见面后，当天 Elon 记录 one-pager
```

避免：两个 Pilot 都在同一天要求紧急决策。

---

### 原则 3：限制「定制化」承诺

最大的 Pilot 杀手是：客户提出一个只有他才需要的功能需求，然后 CEO 说「可以做」。

**原则：** 免费 Pilot 阶段只承诺「标准功能」，任何定制化需求记录 → 进入产品 Backlog → 付费后讨论优先级。

对小语说：「我们先跑标准版，看你用 4 周后哪些功能最重要，再定制。」
对 GC 说：「标准版就是电话接线 + 工单派发，如果有特殊流程，3 周后一起讨论。」

---

### 原则 4：设定 Go/No-Go 里程碑

每个 Pilot 在开始前定好「3 周后的 success criteria」：

**GC Pilot success criteria：**
- [ ] AI 接线 ≥ 50 个电话，客户满意率 ≥ 70%
- [ ] 工单派发准确率 ≥ 85%
- [ ] 客户主动说「这个有用」

**RE 小语 Pilot success criteria：**
- [ ] AI 跟进 ≥ 20 个 buyer leads
- [ ] 小语不需要手动介入 ≥ 60% 的 follow-up
- [ ] 小语觉得「值 $500/月」

如果 3 周后没达标 → 诚实告知，重新 setup，不强推续约。

---

### 原则 5：「Pilot 日志」文化

每个 Pilot 维护一个简单的进展日志：
- 本周发生了什么（客户反馈、数据）
- 下周要做什么（行动）
- 风险/障碍（需要 CEO 决策的）

这个日志是 Elon 的「认知外挂」—— 因为多个 Pilot 并行时，大脑根本记不住所有 context。

---

## 四、CEO 时间分配建议（早期多 Pilot 阶段）

```
白天（执行时间）：
- 70% → 当前最重要的 Pilot（GC 优先级 #1）
- 20% → 第二 Pilot 的关键节点支持
- 10% → 邮件/沟通/admin

夜间（Elon 工作时间）：
- 50% → 研究（GTM/市场/竞品）
- 30% → 产品/技术规划
- 20% → 记忆同步 + 文档整理
```

---

## 五、常见失误 & 对策

| 失误 | 表现 | 对策 |
|------|-----|------|
| 过度承诺 | 对客户说「我们可以加这个功能」 | 先记录，Pilot 结束后讨论 |
| 节奏不一致 | 一周联系 3 次，下周一次都没有 | 设定固定 check-in 日期 |
| 没有量化指标 | 客户说「感觉还好」 → 无法做 Go/No-Go 决策 | 开始前定好 success criteria |
| CEO 事必躬亲 | 两个 Pilot 的所有事都自己做 | 给 Kelvin 清晰的 DRI 角色 |
| 忘记「Pilot 的目的」 | 把 Pilot 当成长期服务在跑 | 3 周硬截止，review 后决策 |

---

## 六、关键洞察

1. **多 Pilot 的本质是「并行假设验证」** —— 每个 Pilot 都是一个实验，不是一个长期承诺
2. **DRI 制度在 AI 团队尤其重要** —— Kelvin 的 context ≠ Elon 的 context，必须明确谁负责哪条线
3. **节奏 > 速度** —— 慢而稳定的 weekly review 比快而混乱的临时响应更能推进 Pilot
4. **「标准功能 Pilot」原则是护城河** —— 它保护你不被每个客户的个性化需求拖垮
