# Maya 产品大战略

> 更新：2026-02-22（v2，补充 Agent 架构 + 隐私设计 + 定价模型）
> 定位：**AI 员工，不是软件**

---

## 核心定位

Maya 是一个 AI 员工，不是 CRM，不是工具，不是 SaaS。

> "给你配一个 AI 员工 Maya，先帮你把客户和项目理清楚，然后越来越多的事情你不用自己做。"

用户和 Maya 的关系是**雇主与员工**，不是用户与软件。

---

## 三圈框架（产品覆盖范围）

```
需求侧（找钱 + 服务）
  └─ 潜在顾客（Leads）→ 现有顾客（Clients）

供应侧（干活）
  └─ 分包商（Subs）、供应商（Suppliers）

内部（自己人）
  └─ 老板、Project Manager、工人
```

### Maya 在每圈做的事

| 圈 | Maya 干什么 |
|---|------------|
| 需求侧 | 跟进 leads、转化报价、更新客户项目进度 |
| 供应侧 | 派活确认、协调日程、催款/发票跟踪 |
| 内部 | 信息中枢、任务分配、状态汇报、知识库 |

---

## 产品阶段

- **Phase 1**：内部（老板 + PM 用得爽）
  - 对话式信息录入，自动整理联系人/项目/待办
  - 老板只需说话，Maya 负责记录和提醒

- **Phase 2**：供应侧（分包商协调）
  - 派活通知、日程确认、变更单捕获
  - 分包商直接和 Maya 短信/电话交互

- **Phase 3**：需求侧（leads → 客户全流程）
  - Lead 跟进自动化
  - 报价生成与追踪
  - 客户项目进度更新

三圈全通 = Maya 是整个 GC 公司的神经系统。

---

## 核心交互模式

**非结构化输入 → 自动结构化输出**

用户像和人说话一样和 Maya 交互，Maya 在后台自动生成：
- 联系人记录
- 项目状态
- 待办任务
- 跟进提醒

用户不需要学任何软件。

## 员工入职逻辑

```
第一周：先听、先学、先记
  → Maya 整理所有联系人、项目、历史记录

第二周：接简单的活
  → 跟进短信、到期提醒、状态更新

第一个月后：真正顶岗
  → 接电话、协调分包商、更新业主
```

越用越依赖——Maya 知道的越多，换掉的成本越高。

## Agent 架构：一个门面，多个专员

**用户视角：** 老板永远只跟 Maya 说话，一个关系，不复杂。

**系统视角：**

```
老板
  ↓ 说话
Maya（主脑 / 总协调）
  ├─ Sub-agent A：内部圈（老板/PM/工人）
  ├─ Sub-agent B：供应侧（分包商/供应商）
  └─ Sub-agent C：需求侧（leads/顾客）
```

Sub-agents 类似 contractor，按需调用，老板无感知。

---

## 隐私设计：信息隔离，不是克制

GC 不想让顾客知道的：材料加价、工地问题、内部争议、真实成本。

解法：**每个 sub-agent 只能访问自己圈层的信息，物理隔离。**

顾客 agent 根本没有成本数据的访问权限——不是 Maya 克制，是看不到。

对外卖点：
> "Maya 足够聪明，知道什么该告诉谁。"

---

## 定价模型：基本工资 + 加班费

GC 天天用这个模型管人，一听就懂。

| 角色 | 类型 | 定价模式 |
|------|------|---------|
| Maya 主脑 | 全职员工 | 月费固定，常驻 |
| 分包商协调 agent | Contractor | 按本月协调次数 |
| 顾客更新 agent | Contractor | 按发出更新条数 |
| Lead 跟进 agent | Contractor | 按跟进 lead 数量 |

**固定底薪保底，多用多付，不用不付。**

与 HWM（Human Working Minutes）模型兼容：
- Maya 月费 = 固定 HWM 额度
- Sub-agent 使用 = 超出额度部分按 HWM 计费

---

## 关键差异化

- 竞品（Procore/BuilderTrend）：软件，需要学习，功能复杂
- Maya：员工，对话交互，越来越懂你的业务
- 定价逻辑：GC 熟悉的雇佣模式，不是陌生的 SaaS 套餐
